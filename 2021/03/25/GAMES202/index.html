Real-Time High Quality Rendering

Real-Time:

Speed: more than 30 FPS, 90FPS for VR/AR

Interactivity: Each frame generated on the fly.(few frames per second, lower than 30)

High Quality:

Realism: advanced approaches to make rendering more realistic

Dependability: all-time correctness (exact or approximate), no tolerance to uncontrollable failures.

What is GAMES202 not about:

  • 3D modeling or game development using Unreal Engine

  • Expensive (but more accurate) light transport techniques in movies / animations (离线渲染，更难)

  • Neural Rendering

    1. 神经网络做的东西一般很难达到实时。

    2. 神经网络很难达到高质量，往往只能做cheery picking, 即挑其中效果最好出来。然而实时是面向用户，无法决定用户显示哪一帧，除非算法能把所有东西覆盖到，而神经网络数据越大越不好训练。

    3. 但是用神经网络降噪是成功的。

  • Using OpenGL

    用 OpenGL shader language来写shader，但是不用OpenGL，他本身是一套不好用的C风格的API.

  • Scene / shader optimization

    纯工程技术，科学知识无变化

  • Reverse performance computing

  • High performance computing e.g. CUDA programming (DX12不能跨平台，比如MAC和OpenGL)

编程人员可以自由编程vertex shader 和 fragment shader

What is OpenGL

OpenGL is a set of APIs that arrange GPU pipeline from CPU(利用CPU调度GPU)

  • Therefore, language does not matter
  • Cross platform
  • Alternatives(DirectX, Vulkan(较新), etc)

Cons:

  • Fragmented: lots of different versions
  • C style, not easy to use
  • Cannot debug (many yeas ago)

1-to-1 mapping to our software rasterizer in GAMES101

How to use OpenGL

Important analogy: oil painting

  • Place objects/models

    • Model specification
      • User specifies an object’s vertices, normals, texture coords and send them to GPU as a Vertex buffer object (VBO)
        • Very similar to .obj files
    • Model transformation
      • Use OpenGL functions to obtain matrices
        • e.g. glTranslate, glMultMatrix, etc.
        • No need to write anything on your own

  • Set position of an easel

    • View transformation
      • Set camera (the viewing transformation matrix) by simply calling, e.g., gluPerspective(GLdouble fovy, GLdouble aspect, GLdouble zNear, GLdouble zFar);
    • Create / use a framebuffer

  • Attach a canvas to the easel

    • Analogy of oil painting:

      • you can also paint multiple pictures using the same easel

    • One rendering pass in OpenGL

      • A framebuffer is specified to use
      • Specify one or more textures as output (shading, depth, etc.) 如果上一帧没渲染完下一帧就开始了，会出现画面撕裂，游戏里的垂直同步就是取消画面撕裂。还有双重缓冲和三重缓冲等多重缓冲，将渲染好的存着，渲染好了才放出来。

  • Paint to the canvas (关键在定义vertex,fragment shaders,其他皆可封装，引擎就相当于GUI)

    • i.e., how to perform shading
    • This is when vertex / fragment shaders will be used

    • For each vertex in parallel

      • OpenGL calls user-specified vertex shader: Transform vertex (ModelView, Projection), other ops

    • For each primitive, OpenGL rasterizes

      • Generates a fragment for each pixel the fragment covers

    • For each fragment in parallel

      • OpenGL calls user-specified fragment shader: Shading and lighting calculations
      • OpenGL handles z-buffer depth test unless overwritten

  • (Attach other canvases to the easel and continue painting)

OpenGL还有立即模式，指定一个三角形渲染一个，然后再指定一个，但是这样非常慢。

  • Multiple passes (Use previous paintings for reference)
    • 比如光线追踪里，要计算一次光线到达的地方，和一次相机看到的地方。

GLSL(OpenGL Shading Languages)

Shadow Mapping

这是一个2-pass的算法，第一个输出light能看到的场景，这叫shadow map，第二个pass从相机渲染整个场景，配合上一个pass输出的light场景，来检测相机看到的每一个点是否能被light看到。

两次计算的距离z要统一，是实际距离或透视投影后的距离。

Pro: not require knowledge of scene’s geometry

Con: causing self occlusion and aliasing issues

自遮挡现象是由于平面像素点内采用同一个光照的深度值，而相机所视点到光源连线大于该点所在像素存储的光照深度(就像是自己挡自己,光源存了一个深度更近的’自己’)。尤其当光源与平面夹角较大时，这一差距更为明显。垂直时较好。

一个解决方法是忽略差距，即设置一个判断条件的偏置，让深度比实际深度小很多才算遮挡。后果是某些细小的深度差被忽略了，导致应有的遮挡没了。工业界中尚无彻底解决的方法，都是设置一个合理的偏置使得适应大多数情况。

有一个方法是Second-depth，

watertight指就算是一张纸也不能只做成一个平面，而是一个很薄的立方体。所以地板就有这个问题。

但是工业界没有多少人用，就算复杂度仍是O(n)，但RTR只看重绝对速度，它仍然是花了两倍的时间资源。RTR也不像机器学习那样考虑误差上下限，比较偏实际应用。

在30fps的要求中，每帧里每个效果的渲染只能占据1ms的时间资源。

Approximation calculas in RTR

微积分中有很多不等式，在实时渲染里近似使用不等式为约等式。

当积分域(support)很小的时候，或者$g(x)$在积分域下变化不大的时候，这个近似较为准确。

这样，就把visibility从shader中分离了出来，让shader不用考虑visibility。

当积分域很小(small support)的时候是准的，在这里就是一个点光源的时候，光线面积只有一个点，或者方向(directional)光源，只有一个方向有光。

或者当积分函数变化不大的时候(smooth integrand)，$L_i$也不变，BRDF是diffuse的时候。

所以当存在环境光照或者面光源的时候，以及glossy的BRDF(变化不平滑，比较剧烈)的时候，不适合用shadow mapping，因为不精确，当然强行用也是可以。

在Ambient Occlusion还会用到。

Percentage closer soft shadows (PCSS) 很古老但许多地方仍是SOTA。其核心是适应性的filter size加上PCF

现实生活中见到的大多数光源是面光源，所以软阴影看起来比较自然。

实现软阴影，首先要用到一个工具：Percentage Closer FIltering(PCF)

最早是用来做阴影边界的抗锯齿的，后来被发现用作软阴影。

它是类似卷积那样，把一个点周围一个区域的像素的visibility做平均 (开销解决在PCSS中说)。得到的visibility就不再是0-1二值，而是介于其中的一个小数(这个小数可视为这个区域里有百分之多少的像素深度值不深于shadow map)。它不是filter最终的锯齿结果(模糊锯齿)，也不是filter shadow map的深度。

也就是对判断是否遮挡的visibility做filter。

当filter很大的时候，就相当于把大片的阴影给模糊了，就实现了软阴影的效果。但是具体filter要多大更好呢？

物体离地板近的阴影更硬，远的更软，所以距离决定了filter大小，就是下图中blocker和receiver的距离。通过相似三角形得出半影(penumbra)的大小，这就是filter核的大小。

可以从shading point与光源连线的范围里去找遮挡物，及其平均深度是多少，然后就可以知道filter核是多大。

开销很大，如果是多光源，shadow mapping只能一个个处理，多几个光源就多几倍开销。

Variance soft shadow mapping(为了解决PCSS中第1和3步较慢的问题，Blocker search和PCF)

第三步：

PCF对每一个点计算的实质是：得到一个区域里有百分之多少的像素深度值浅于shadow map，也就是在一个分布中找到某一个点的排位，

如果将深度的分布近似成一个正态分布，就只需要知道均值和方差，就可以找到某个x在所有值里的大小位置。

均值可由mipmap得到(但是不准，下面有更准的做法)，方差可在生成shadow map的时候一起生成一个深度的平方的图，然后和均值共同计算出来。

但是这一分布的CDF(累计分布函数)，可能没有解析解，毕竟不是真正的正态分布。C++里有erf函数可以计算数值解。

但是VSSM其实并不需要精确的解析解，可以通过切比雪夫不等式来进行近似。这个不等式可以知道一个随机变量取一个值超过某一个值的概率，并且不需要知道确切的分布函数(甚至可以不是正态分布，只是隐含地说这是一个单峰的分布，很像正态分布)，只需要均值(期望)和方差。

但是有一个条件，这个t必须大于均值，如果小于则不准。但是人们一直都在用，还是好用。实时渲染里大家用了太多假设。

某一个点，通过周围一块区域深度的均值和方差，可以计算得出：该点深度大于某一个值(shadow map中存储深度)的概率，用1减去这一概率就得到该点深度小于shadow map深度的概率，即这一块区域有百分之多少没有被挡住，那么这一点的可见性就是多少。

有物体移动或者光源移动，就要重新计算一遍mipmap。但是mipmap的GPU硬件支持十分到位，几乎不需要时间。

第一步：

有很多邻域均值的计算操作，第一步中有找一块区域里遮挡物的深度均值，如果累加每个深度再除总数，效率太低。

可以仅使用mipmap的范围查询和切比雪夫不等式来得出这一深度均值。

这里有一个更大胆的假设，非遮挡物的深度全都等同于当前点的深度。最终$z_{occ}$就可通过计算得出。

VSSM是十分聪明的做法，更快且没有任何噪声，不过现在很多人用PCSS，它的效果挺好可以作为一个baseline，而且现在对于噪声容忍度很高，因为降噪的方法很好用，特别是基于temporal的方法，比如PAA。所以较快地得出有噪声的结果是很好的。

另外，VSSM在平面的光接受面效果好，如果背景是曲面，或者垂直于光源，就会出现问题。

比mipmap更准的范围查询算法——前缀和

使用一个特殊的存储结构——SAT

一维：每个元素存的是当前值加上以前所有值的累积。

二维：每个矩形的起点是左上角，每个左下角都是这个矩阵从左上角累加到左下角的值。只需每行做一次一维的SAT，然后每列也做一次。

累加的数据精度会差一点。行与行还是可以并行处理SAT的。 如果有新的shadow map也需要计算新的SAT

Moment shadow mapping

实现起来有点麻烦，被PCSS逐渐压过。VSSM在分布的假设上会出现不准确，如果实际分布是多峰的，百分比估计会出现两种问题——过黑(尚可接受)，或者在阴影之间出现白色(可视性百分比偏高,light leaking或light bleeding)。

还有接受面为曲面的时候，切比雪夫不等式的条件。

MSM只解决分布不准确的问题，通过使用均值的更高阶的moments(矩，定义很多，最简单的是次幂)可以更好地拟合函数。保留前m阶矩，就可以表示出有$\frac{m}{2}$个台阶的函数。比如下图，只用前两阶就是VSSM。

越多拟合得越真实，但是存储会变大，通常存四阶的就够了。工业界里经常用packing和unpacking来节约存储，比如32位的数砍一半变16位，然后两个拼在一起当一个数来存。不过这样自定义了一个数，GPU就不能自动插值。

用这四阶如何拟合分布函数是非常复杂的，只能自己去看论文《Moment shadow mapping》。

Lecture 5-6 Real-time Environment Mapping

PRT(precomputed radiance transfer)

In real time rendering it’s very hard to get shadow from environment lighting：

  • Different perspectives of view

    • As a many-light problem Cost of shadow map is linearly to light
    • As a sampling problem Visibility term can be arbitrarily complex and easily separated from environment

  • Industrial solution

    • Generate one (or a bit more) shadows from the brightest light sources

  • Related research

    • Imperfect shadow maps
    • Light cuts(clarify lights)
    • RTRT—real time ray tracing=path tracing + denosie(might be the ultimate solution)
    • Precomputed radiance transfer

Background knowledge

  • Frequency and filtering

Any product integral can be considered as filtering

The frequency of the integral is the lowest of any individual’s.

  • Basis funuctions

A set of functions that can be used to represent other functions in general

The Fourier series is a set of basis functions

The polynomial series(1,$x$,$x^2$,…) can also be a set of basis functions

Spherical Harmonics

It’s a set of 2D basis functions defined on the sphere

Analogous to Fourier series in 1D

图中颜色越亮的地方表明绝对值越大，蓝色是正，黄色是负，位置则是坐标

The Spherical Harmonics is associated with Legendre polynomial(勒让德函数)

$B_i(w)$ is SH basis functions, $f(w)$ is any functions.

This multiple integral is a kind of Projection. The $f(w)$ can be obtained by $B_i(w)$ and $c_i$.

阶数 $l$ 越高，频率越高越精细

任何材质，只要是diffuse的，都可以用前三阶的SH来描述光照，然后来照物体做shading。因为diffuse的BRDF本来就只有低频。(思路是先分析BRDF的频率，再考虑光照)

因此，SH比较适合用来描述低频的。

接下来做shadow，就使用PRT

人们喜欢把Rendering Equation写成triple product的形式，lighting、visibility、BRDF(包括cos或max)

三个项都可以写成球面函数(BRDF的出射、入射角度确定)，然后对每个shading point，都把所有像素相乘然后加起来，比如一个块像素是64x64，6个块组成环境，就是6x64x64。这样计算太复杂，那么使用precomputed radiance transfer可以大大减少计算复杂度。

PRT的思想是，假设环境中只有光会变化，其他的都不变，于是可以预计算。

rendering equation分成两部分，光和剩下的transport部分都可以看视为一系列基函数的线性组合。第二部分确定一个出射光方向o，对每个入射方向i进行积分，所以还是个球面函数。

在diffuse情况下，BRDF就是个常数$k_d$，因此可以写成

然后把$L(i)$直接拆分为基函数的线性组合，然后把$B_i(i)$给另一部分，光源就可以只用一个$l_i$向量来表示。把光源的$B_i(i)$给另一部分，就相当于用光分解出的基函数去照物体

这样，积分内就变成了一个球面函数在基函数上的投影，就是一些具体数值，因此可以进行预计算

积分内就变成了对于每一个i，乘积的加和，就相当于两个向量的点乘。$l_i$也是预计算的，用$\int L(i)B(i)$就可以求出

这样预计算假设了除了光源其他都不变，即场景不能发生改变或动态。光源的旋转也可以解决，因为球谐函数具有旋转不变性，光源转动也能够立刻得到对应的系数$l_i$是多少。$V(i)$不会改变，因为$V$是指从一个点向四面八方看有没有其他东西挡住他。

旋转光(或一个图)，就相当于旋转它的SH basis，而SH旋转后，可以通过同阶的其他SH basis线性组合而得。这就可以打一张表预计算。
